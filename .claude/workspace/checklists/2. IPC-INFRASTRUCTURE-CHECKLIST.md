## PHASE 1: Core Infrastructure Modules

**State Transformation**: Scattered IPC calls with no centralization → Modular IPC primitives with single responsibility

**Narrative**:
This phase establishes the atomic building blocks for unified IPC management. Each module handles one concern: tracking, caching, batching, and metrics. These primitives create the foundation for Phase 2's manager assembly and enable Phase 3's Engine integration.

**Dependencies** (3D Graph):

- **Sequential** (Z-axis): None (entry phase)
- **Lateral** (X-axis): Existing `ipc-performance-tracker.ts` provides reference implementation
- **Diagonal** (Y-axis): None

**Verification**: `npx tsc --noEmit src/renderer/engine/ipc/*.ts`

## PHASE 4: Main Process Batch Support

**State Transformation**: Main process handles individual IPC calls → Batch handler processes grouped calls

**Narrative**:
This phase adds main process support for batched IPC calls from the renderer. The `IPC_BATCH` channel receives grouped calls and dispatches them internally, returning aggregated results. This completes the batching loop started in Phase 1.

**Dependencies** (3D Graph):

- **Sequential** (Z-axis): Phase 3 complete (Engine.ipc available)
- **Lateral** (X-axis): Existing IPC handler patterns in main process
- **Diagonal** (Y-axis): Task 1.3 (batch queue) output shapes batch format

**Verification**: `npm run build && npm run test:ipc`
