```
│                         RENDERER PROCESS                         │

│  │        └──────────────┴──────────────┴──────────────┘        │ │
│  │                              │                                │ │
│  │                              ▼                                │ │
│  │                      session.db (WAL)                        │ │
│  └─────────────────────────────────────────────────────────────┘ │
│                         MAIN PROCESS                             │
└──────────────────────────────────────────────────────────────────┘
```

**Core Principles**:
- Monaco is authoritative for active buffer content
- DB is queryable snapshot, updated via debounced sync
- Single session.db with tab_id foreign keys (not per-tab DBs)
- All renderer access via Engine.ipc → SessionDbService
- **One-way writes**: Renderer pushes snapshots, never reads them directly
- **Query-only reads**: Renderer reads exclusively via SessionQuery API

## Architectural Invariants

### One-Way Snapshot Writes

```
┌──────────────────────────────────────────────────────────────────┐
│  RENDERER                                                         │
│  ┌─────────────┐                                                  │
│  │   Monaco    │──pushSnapshot()──▶ IPC ──▶ DB (write-only)      │
│  │  (authority)│                                                  │
│  │             │◀──query()◀─────── SessionQuery ◀── DB (read)    │
│  └─────────────┘                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**Invariant**: The renderer NEVER directly reads buffer_snapshots or buffer_lines.
- Write path: `pushBufferSnapshot()` - fire-and-forget to main process
- Read path: `Engine.session.query()` - always via SessionQuery abstraction

**Why this matters**: SessionQuery can later be moved to:
- A Web Worker (off main thread)
- A WASM module (for complex queries)
- In-process renderer (if we add SQLite WASM)
- Remote service (for cloud sync)

The one-way write contract ensures no coupling to storage location.

### Hot vs Cold Query Paths

```typescript
interface QueryOptions {
    /** Cold query: bypasses cache, can be throttled/deprioritized */
    cold?: boolean;
    /** Cancel signal for long-running queries */
    signal?: AbortSignal;
}

// Hot queries - must be fast, cached aggressively
Engine.session.query('SYMBOL_AT_CURSOR', { line, col });           // <5ms
Engine.session.query('ACTIVE_BUFFER', {});                          // <2ms

// Cold queries - can be slow, run in background
Engine.session.query('PROJECT_SEARCH', { pattern }, { cold: true }); // async
Engine.session.query('FULL_REINDEX', {}, { cold: true });            // background
```

**Hot queries** (default):
- Use cache aggressively
- High priority in IPC queue
- Timeout quickly (50ms)
- Examples: cursor symbol, active buffer, visible viewport

**Cold queries** (`{ cold: true }`):
- Bypass cache (always fresh)
- Low priority, can be throttled
- No timeout (use AbortSignal)
- Examples: project-wide search, reindex, import graph traversal

### Buffer Epoch for Cross-Restart Comparison

The `buffer_epoch` column is a monotonically increasing counter that:
1. Increments on each snapshot write
2. Persists across app restarts
3. Enables comparison: "is this snapshot newer than that one?"

```sql
-- buffer_epoch is stored in a separate table for atomicity
CREATE TABLE session_meta (
    key TEXT PRIMARY KEY,
    value INTEGER NOT NULL
);
-- Initialize: INSERT INTO session_meta VALUES ('buffer_epoch', 0);

-- On each snapshot write:
-- 1. UPDATE session_meta SET value = value + 1 WHERE key = 'buffer_epoch'
-- 2. SELECT value FROM session_meta WHERE key = 'buffer_epoch'
-- 3. INSERT INTO buffer_snapshots (..., buffer_epoch) VALUES (..., epoch)
```

This enables:
- Conflict resolution on restore
- Detecting stale snapshots after crash recovery
- Ordering snapshots from different tabs temporally

```sql
-- Full schema for session.db

-- Session Metadata (monotonic counters, global state)
CREATE TABLE session_meta (
    key TEXT PRIMARY KEY,
    value INTEGER NOT NULL
);
-- Initialize buffer_epoch counter (monotonic across restarts)
INSERT INTO session_meta (key, value) VALUES ('buffer_epoch', 0);

-- Tab Management
CREATE TABLE open_tabs (
    tab_id TEXT PRIMARY KEY,
    file_path TEXT NOT NULL,
    is_dirty INTEGER DEFAULT 0,
    is_active INTEGER DEFAULT 0,
    cursor_line INTEGER DEFAULT 1,
    cursor_column INTEGER DEFAULT 1,
    scroll_top INTEGER DEFAULT 0,
    viewport_start_line INTEGER DEFAULT 1,
    viewport_end_line INTEGER DEFAULT 50,
    opened_at INTEGER NOT NULL,
    last_focused_at INTEGER,
    UNIQUE(file_path)
);

-- Buffer Content (snapshot-based)
CREATE TABLE buffer_snapshots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    tab_id TEXT NOT NULL,
    version INTEGER NOT NULL,
    buffer_epoch INTEGER NOT NULL,
    content TEXT NOT NULL,
    line_count INTEGER NOT NULL,
    byte_size INTEGER NOT NULL,
    snapshot_at INTEGER NOT NULL,
    FOREIGN KEY (tab_id) REFERENCES open_tabs(tab_id) ON DELETE CASCADE
);
-- buffer_epoch: Monotonic counter for cross-restart ordering
-- Increment atomically: UPDATE session_meta SET value = value + 1 WHERE key = 'buffer_epoch'

-- Buffer Content (line-based for large files)
CREATE TABLE buffer_lines (
    tab_id TEXT NOT NULL,
    line_number INTEGER NOT NULL,
    content TEXT NOT NULL,
    PRIMARY KEY (tab_id, line_number),
    FOREIGN KEY (tab_id) REFERENCES open_tabs(tab_id) ON DELETE CASCADE
);
```

## Appendix: Query Examples

```typescript
// =============================================================================
// HOT QUERIES (default) - fast, cached, for UI responsiveness
// =============================================================================

// Tab operations (always hot - UI needs instant response)
const tabs = await Engine.session.getTabs();
const activeTab = await Engine.session.getActiveTab();
await Engine.session.openTab('/path/to/file.ts');
await Engine.session.closeTab(tabId);

// Buffer operations (hot - active editing)
const content = await Engine.session.getActiveBuffer();
const lines = await Engine.session.getBufferLines(tabId, 1, 100);

// Symbol at cursor (hot - must be <5ms for hover/autocomplete)
const symbolAtCursor = await Engine.session.getSymbolAtPosition(filePath, line, col);

// =============================================================================
// COLD QUERIES ({ cold: true }) - fresh, uncached, can be slow
// =============================================================================

// Project-wide search (cold - user expects delay)
const results = await Engine.session.searchAll('TODO:', { cold: true });

// Full symbol search (cold - can scan entire index)
const allSymbols = await Engine.session.findSymbol('use*', {
    cold: true,
    signal: abortController.signal  // Allow cancellation
});

// Import graph traversal (cold - can be expensive)
const imports = await Engine.session.getFileImports('/src/main.ts', { cold: true });
const dependents = await Engine.session.getFileExporters('/src/utils.ts', { cold: true });

// Reindex operation (cold - background task)
await Engine.session.query('FULL_REINDEX', {}, { cold: true });

// =============================================================================
// ONE-WAY WRITES (fire-and-forget from renderer)
// =============================================================================

// Push snapshot - renderer NEVER reads this back directly
Engine.ipc.send('SESSION_BUFFER_PUSH_SNAPSHOT', { tabId, content, version });

// Push lines for large files
Engine.ipc.send('SESSION_BUFFER_PUSH_LINES', { tabId, lines });

// Note: To READ buffer content, use SessionQuery:
const restored = await Engine.session.getActiveBuffer();  // Via query, not direct read
```

## Appendix: Hot vs Cold Query Classification

| Query Type | Default | Reason |
|------------|---------|--------|
| `getTabs()` | Hot | Tab bar needs instant render |
| `getActiveBuffer()` | Hot | Editor restore must be fast |
| `getSymbolAtPosition()` | Hot | Hover/autocomplete latency sensitive |
| `findSymbol()` | Configurable | Quick lookup = hot, wildcard search = cold |
| `searchAll()` | Cold | Project-wide, user expects progress indicator |
| `getFileImports()` | Cold | Graph traversal, can be expensive |
| `FULL_REINDEX` | Cold | Background operation |
